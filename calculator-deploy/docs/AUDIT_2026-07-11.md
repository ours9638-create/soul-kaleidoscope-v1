日期：2026-07-11  
版本基準：2.2.0

- 正式 Worker：`soul-kaleidoscope-v1`

- 國曆轉農曆資料由專案內建 1940～2035 對照表產生。
- 農曆資料以 `globalThis.LUNAR_DATA` 提供給瀏覽器計算引擎。
- 凌晨 `00:xx` 以 24 計算。
- 國曆與農曆生日狀態分開判定，並在同一結果格中顯示。
- 農曆跨國曆年生日可正確定位。
- `37/10/1` 依專案確認口徑判為 5級。
- 手機版首頁、輸入卡片與結果卡片使用同一外層寬度。
- GitHub 與 Cloudflare 已連線，自動部署不再更換正式網址。
- 核心引擎包含 27 項回歸測試。

## 三、本次稽核發現與處理

### 1. 公式來源存在衝突

原始整合試算表與後續確認規則有兩項差異：

- 農曆生日狀態的年度定位方式不同。
- `37/10/1` 靈魂等級為 4級或 5級的口徑不同。

處理：

- `RULES_LOCK.md` 明定使用者確認規則優先。
- 建置時執行固定案例回歸測試。
- 整合試算表後續必須同步修正，未修正前不得作為唯一真值來源。

### 2. 正式來源檔與部署內容曾不一致

原始 `index.html`、`script.js`、`sw.js` 曾保留舊版內容，依賴 `build-calculator.js` 在 Cloudflare 建置時臨時改寫：

- 注入 `layout-fix.css`。
- 合併國曆／農曆生日狀態。
- 修改 DOM ID。
- 修改前端 JavaScript。
- 修改頁面版號。
- 修改 Service Worker cache name。

風險：

- GitHub 直接閱讀的原始碼不是實際部署碼。
- 字串替換失敗時，頁面與程式可能使用不同 DOM ID。
- 版本與快取可能不同步。
- 本機、Cloudflare 組建與正式網站難以追溯。

已處理：

- `index.html` 直接保存正式版面、版本與合併生日狀態。
- `script.js` 直接使用 `summaryBirthdayStatus`。
- `sw.js` 直接保存正式 cache 版本。
- 建置只生成 `lunar-data.js`，不再改寫 HTML、JS、CSS 或 Service Worker。

### 3. 前端初始化失敗時缺少診斷

原本 `LUNAR_DATA` 或核心引擎載入失敗時，程式可能在事件綁定前停止，使用者只看到按鈕沒有反應。

已處理：

- 初始化前檢查核心物件與農曆資料。
- 失敗時在頁首顯示「引擎載入失敗」與處理建議。
- 成功時顯示 27 項自檢結果。

### 4. 建置門檻不足以檢查來源一致性

原本只執行公式回歸，未檢查：

- 頁面版號。
- Service Worker cache 版本。
- 合併生日狀態 DOM。
- 前端是否仍引用舊 ID。
- 必要檔案是否存在。
- 前端 JavaScript 是否可解析。

已處理：

- 新增 7 項靜態來源驗證。
- 新增核心、套件、頁面與快取版本一致性檢查。
- 新增前端 JavaScript 語法檢查。
- 任一失敗即阻止 Cloudflare 部署。

### 5. PWA 快取來源未直接同步

來源 `sw.js` 曾停留在 2.1.3，建置時才修改成 2.2.0。

已處理：

- 來源檔直接改為 2.2.0。
- 前端註冊完成後主動呼叫 `registration.update()`。
- Service Worker 只處理同源 GET 請求。

### 6. P1 資料保存範圍需要先鎖定

若直接儲存完整計算結果，未來公式修正後會保留過期結果。

決策：

- P1 只保存輸入資料與版本資訊。
- 載入後由目前引擎重新計算。
- 每筆使用 UUID，不以姓名或生日作為唯一鍵。
- JSON 匯入先完整驗證，失敗時不得改寫既有資料。
- 第一版使用 `localStorage`，達到容量或複雜度門檻後才升級 IndexedDB。

詳細規格：`P1_STORAGE_SPEC.md`。
