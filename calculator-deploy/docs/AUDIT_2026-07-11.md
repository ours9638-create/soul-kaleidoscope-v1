# 靈魂萬花筒計算器｜技術稽核

日期：2026-07-11  
版本基準：2.2.0

## 一、目前架構

- 正式 Worker：`soul-kaleidoscope-v1`
- GitHub 儲存庫：`ours9638-create/soul-kaleidoscope-v1`
- 生產分支：`calculator-deploy-setup`
- 根目錄：`calculator-deploy`
- Cloudflare 組建：`npm run build`
- Cloudflare 部署：`npx wrangler deploy`
- 網址維持固定，由 GitHub commit 自動觸發部署。

## 二、已確認正常

- 國曆轉農曆資料由專案內建 1940～2035 對照表產生。
- 農曆資料以 `globalThis.LUNAR_DATA` 提供給瀏覽器計算引擎。
- 凌晨 `00:xx` 以 24 計算。
- 國曆與農曆生日狀態分開判定。
- 農曆跨國曆年生日可正確定位。
- `37/10/1` 依專案確認口徑判為 5級。
- 手機版首頁、輸入卡片與結果卡片使用同一外層寬度。
- GitHub 與 Cloudflare 已連線，自動部署不再更換正式網址。

## 三、稽核發現

### 1. 公式來源存在衝突

原始整合試算表與後續確認規則有兩項差異：

- 農曆生日狀態的年度定位方式不同。
- `37/10/1` 靈魂等級為 4級或 5級的口徑不同。

處理：新增 `RULES_LOCK.md`，明定使用者確認規則優先。

### 2. P0 原先不能稱為完全完成

先前瀏覽器僅有 4 項自檢，無法涵蓋主要公式。

處理：2.2.0 擴充為 27 項核心回歸測試，並加入 Cloudflare 建置門檻。任何測試失敗，部署直接中止。

### 3. 建置流程曾過度依賴字串替換

目前組建腳本仍會處理版號、生日狀態合併與快取名稱。此方式可運作，但維護成本高。

建議：下一個整理版將最終 HTML/JS 結構直接寫回來源檔，建置腳本只負責產生農曆資料、執行測試與部署前檢查。

### 4. 缺少正式資料保存策略

目前輸入資料僅存在頁面記憶體，重新整理即消失。

建議：P1 先採瀏覽器本機儲存，不立即使用雲端資料庫，避免姓名、生日與出生時間直接上雲。

### 5. 缺少報告版本控制

未來產出 PDF、解讀文字與靈魂萬花圖後，必須記錄：

- 計算引擎版本
- 規則版本
- 解讀資料庫版本
- 報告產生時間

否則舊報告無法追溯當時使用的口徑。

## 四、風險分級

| 風險 | 等級 | 處理方式 |
|---|---:|---|
| 公式來源衝突 | 高 | 規則鎖定＋回歸測試 |
| 農曆跨年判斷 | 高 | 固定案例測試 |
| PWA 舊快取 | 中 | 每版更新 cache name 與可見版號 |
| 個案資料遺失 | 中 | P1 本機儲存與 JSON 備份 |
| 個資上雲 | 高 | P5 前不啟用雲端同步 |
| 報告內容口徑漂移 | 高 | 資料庫版本與報告版本綁定 |
| 手機表格過寬 | 低 | 卡片全寬＋表格區域獨立橫向捲動 |

## 五、P0 完成定義

P0 只有在以下條件同時成立時才算完成：

- 已確認公式全部寫入 `RULES_LOCK.md`。
