# P1｜個案本機儲存、搜尋與備份技術規格

版本：`1.1`  
目標 App：`2.3.x`  
狀態：第一版程式完成，等待 Cloudflare 與 iPhone 實機驗收

## 1. 目的

在不建立帳號、不使用雲端資料庫的前提下，讓使用者可以在目前瀏覽器：

- 儲存個案。
- 搜尋與載入。
- 覆寫、另存與刪除。
- 匯出 JSON 備份。
- 匯入並合併備份。

本階段只永久保存輸入資料與必要版本資訊。載入個案後，使用目前正式引擎重新計算，不永久保存可能已過期的完整計算結果。

## 2. 儲存位置

第一版使用瀏覽器 `localStorage`。

儲存鍵：

```text
soul-kaleidoscope.case-store
```

適用範圍：

- 約 100～500 筆純文字個案。
- 離線使用。
- 不需要帳號與伺服器。

升級至 IndexedDB 的條件：

- 個案明顯超過 500 筆。
- 需要保存圖片、PDF 或大量報告草稿。
- 需要複合索引、交易或更大容量。

## 3. 版本分離

目前：

- App Version：`2.3.0`
- Engine Version：`2.2.0`
- Case Store Module：`1.0.0`
- Case Store Schema：`1`

App 功能更新不代表公式更新。每筆個案記錄保存建立時的引擎版本；載入後仍使用目前引擎重算。

## 4. 資料庫格式

```json
{
  "schemaVersion": 1,
  "exportedAt": "2026-07-12T12:00:00.000Z",
  "appVersion": "2.3.0",
  "records": []
}
```

單一個案：

```json
{
  "id": "uuid",
  "name": "個案姓名",
  "solarBirth": "1989-05-28",
  "birthTime": "15:17",
  "queryDate": "2026-07-05",
  "lunarBirth": {
    "year": 1989,
    "month": 4,
    "day": 24,
    "leap": false
  },
  "engineVersion": "2.2.0",
  "createdAt": "2026-07-12T12:00:00.000Z",
  "modifiedAt": "2026-07-12T12:00:00.000Z"
}
```

## 5. 不永久保存

- 完整計算結果。
- SNGL 報告全文。
- PDF、圖片或長圖。
- 剪貼簿內容。
- Cloudflare、GitHub 或第三方識別資料。

理由：公式與內容資料庫會持續校正。載入後重算可確保使用目前正式規則。

## 6. 個案 ID

- 優先使用 `crypto.randomUUID()`。
- 不以姓名、生日或時間組成 ID。
- 同名個案可並存。
- 「儲存為新個案」一定建立新 ID。
- 「覆寫目前個案」保留原 ID 與 `createdAt`。

## 7. 操作流程

### 7.1 儲存為新個案

1. 重新執行表單驗證與核心計算。
2. 建立最新 Soul Profile。
3. 建立新 UUID。
4. 保存輸入、引擎版本與時間戳。
5. 更新列表並選取新紀錄。

### 7.2 覆寫

1. 必須先載入一筆個案。
2. 再次驗證與重新計算。
3. 保留 ID 與 `createdAt`。
4. 更新輸入、引擎版本與 `modifiedAt`。

### 7.3 載入

1. 點選個案列表。
2. 將保存的輸入填入表單。
3. 重新檢查農曆與閏月。
4. 使用目前核心引擎重新計算。
5. 顯示該紀錄建立時的引擎版本。

### 7.4 刪除

- 必須先選取個案。
- 顯示姓名與國曆生日確認。
- 取消時不得刪除。
- 完成後更新列表、筆數與選取狀態。

### 7.5 搜尋與排序

- 搜尋姓名片段。
- 搜尋完整生日或 `YYYY-MM`。
- 預設依 `modifiedAt` 由新到舊。

## 8. JSON 匯出

檔名：

```text
soul-kaleidoscope-cases-YYYY-MM-DD.json
```

匯出前顯示：

- 個案筆數。
- 明文個資警示。

匯出後保存最後備份時間，但不保存備份檔本身。

## 9. JSON 匯入

第一版只提供「合併」，不提供直接清空取代。

匯入前：

1. 解析 JSON。
2. 遷移舊 Schema。
3. 驗證所有紀錄。
4. 計算預覽：
   - 新增。
   - 更新。
   - 相同略過。
   - 保留本機較新資料。
5. 使用者確認後才寫入。

合併規則：

- ID 不存在：新增。
- ID 相同且內容完全相同：略過。
- ID 相同且匯入檔 `modifiedAt` 較新：更新。
- ID 相同且本機較新或相同：保留本機並列為衝突。

任何解析、格式或 Schema 錯誤都不得先寫入本機資料。

## 10. 資料遷移

目前支援：

- Schema `1`：直接使用。
- Schema `0`：遷移至 Schema `1`。
- 舊版資料直接為陣列：包裝成 Schema `1`。
- 舊欄位 `solarBirthDate`：遷移為 `solarBirth`。

不支援的未來 Schema 必須拒絕，不可自行猜測。

## 11. 隱私與安全

- 所有資料預設只存在目前瀏覽器。
- 不放入 URL、分析事件、console 或遠端錯誤日誌。
- GitHub 只保存假資料測試案例。
- JSON 備份為明文，畫面必須清楚警示。
- 清除 Safari 網站資料或移除 PWA 前提醒先備份。
- 私密瀏覽模式不保證長期保存。
- P5 才評估加密與跨裝置同步。

## 12. 介面

「個案管理（僅此裝置）」卡片包含：

- 搜尋框。
- 本機筆數。
- 目前個案。
- 最後備份時間。
- 個案列表。
- 儲存為新個案。
- 覆寫目前個案。
- 刪除目前個案。
- 匯出全部備份。
- 匯入並合併。
- 本機保存與明文備份警示。

手機版不得產生整頁水平捲動。

## 13. 自動測試

執行：

```bash
npm run test:cases
```

目前 20 項檢查涵蓋：

- 空資料庫與 Schema。
- 新增、同名、覆寫與刪除。
- 搜尋與排序。
- 匯出／匯入一致。
- 無效 JSON 與不支援 Schema。
- 匯入失敗不改變原資料。
- 100 筆保存與搜尋。
- 舊 Schema 與舊欄位遷移。

## 14. 人工驗收

- Safari 關閉再開後資料仍存在。
- 主畫面 PWA 關閉再開後資料仍存在。
- iPhone 可在「檔案」App 找到備份。
- 匯出後刪除，再匯入可完整還原。
- 錯誤檔案不破壞既有資料。
- 100 筆列表與搜尋仍順暢。
- LocalStorage 不可用時，計算功能仍正常，個案管理顯示錯誤並停用。
