# P1｜個案本機儲存、搜尋與備份技術規格

版本：草案 1.0  
目標產品版本：2.3.x

## 1. 目的

在不建立帳號、不使用雲端資料庫的前提下，讓使用者可以在目前瀏覽器保存個案、搜尋、載入、覆寫、另存、刪除，並以 JSON 完成備份與還原。

本階段只處理個案輸入資料與必要的版本資訊。正式計算結果在載入個案後，由目前版本的核心引擎重新計算，避免長期保存已過期或已修正前的錯誤結果。

## 2. 儲存位置

第一版使用瀏覽器 `localStorage`。

理由：

- 實作簡單、離線可用。
- 100～500 筆純文字個案資料容量足夠。
- 不需要伺服器與登入。
- 適合先驗證實際使用流程。

升級條件：

- 個案數量明顯超過 500 筆。
- 需要保存圖片、PDF 或大量報告草稿。
- 需要複合索引或更完整交易機制。

達到上述條件後再遷移至 IndexedDB，不在 P1 第一版提前增加複雜度。

## 3. 儲存鍵與資料庫格式

儲存鍵：

```text
soul-kaleidoscope.case-store
```

資料庫外層：

```json
{
  "schemaVersion": 1,
  "exportedAt": "2026-07-11T12:00:00.000Z",
  "appVersion": "2.3.0",
  "records": []
}
```

單一個案紀錄：

```json
{
  "id": "uuid",
  "name": "個案姓名",
  "solarBirth": "1989-05-28",
  "birthTime": "15:17",
  "queryDate": "2026-07-05",
  "lunarBirth": {
    "year": 1989,
    "month": 4,
    "day": 24,
    "leap": false
  },
  "engineVersion": "2.3.0",
  "createdAt": "2026-07-11T12:00:00.000Z",
  "modifiedAt": "2026-07-11T12:00:00.000Z"
}
```

## 4. 不儲存的內容

P1 不永久保存：

- 目前頁面的完整計算結果。
- PDF 或長圖檔案。
- 解讀文章全文。
- 剪貼簿內容。
- Cloudflare、GitHub 或任何第三方識別資料。

理由：公式與解讀資料會持續校正。載入後重新計算，才能保證使用的是目前正式規則。

## 5. 個案 ID

- 使用 `crypto.randomUUID()`。
- 不使用姓名、生日或時間組成 ID。
- 同名個案可以並存。
- 「另存」一定建立新 ID。
- 「覆寫」只更新目前已載入的 ID。

## 6. 操作流程

### 6.1 儲存為新個案

1. 檢查國曆生日、出生時間、查詢日期、農曆年月日完整。
2. 執行一次核心計算，確認輸入有效。
3. 建立新 ID。
4. 保存輸入資料、引擎版本、建立時間與修改時間。
5. 重新整理列表，選取新紀錄。

### 6.2 覆寫目前個案

1. 必須已有目前個案 ID。
2. 再次執行輸入驗證與核心計算。
3. 保留 `createdAt`。
4. 更新輸入資料、`engineVersion`、`modifiedAt`。

### 6.3 載入

1. 從列表選擇個案。
2. 將輸入資料填入表單。
3. 重新執行農曆與閏月檢查。
4. 以目前核心引擎重新計算。
5. 顯示「此紀錄原建立版本」與「目前重新計算版本」。

### 6.4 刪除

- 顯示姓名與生日確認。
- 必須再次確認。
- 刪除後不可直接復原，提醒先匯出備份。

### 6.5 搜尋與排序

- 搜尋欄比對姓名、國曆生日。
- 預設依 `modifiedAt` 由新到舊排序。
- 列表至少顯示姓名、國曆生日、最後修改時間。

## 7. JSON 匯出

檔名：

```text
soul-kaleidoscope-cases-YYYY-MM-DD.json
```

匯出內容：

- `schemaVersion`
- `appVersion`
- `exportedAt`
- 全部個案紀錄

匯出前顯示：

- 個案筆數
- 目前版本
- 備份檔包含姓名、生日與出生時間，請妥善保存

## 8. JSON 匯入

匯入前檢查：

- 必須是有效 JSON。
- `schemaVersion` 必須可支援。
- `records` 必須是陣列。
- 每筆資料的日期與時間格式必須有效。
- 任何錯誤都不可先寫入本機資料。

匯入模式：

### 合併

- ID 不存在：新增。
- ID 已存在且匯入檔較新：顯示衝突清單，再決定是否覆寫。
- ID 已存在且內容相同：略過。

### 取代

- 必須二次確認。
- 先產生目前資料的自動備份下載。
- 完成備份後才可清空並匯入。

第一版預設使用「合併」，不預設取代。

## 9. 資料遷移

每次讀取資料庫時：

1. 檢查 `schemaVersion`。
2. 版本相同：直接使用。
3. 舊版本：先在記憶體中遷移並驗證。
4. 遷移成功後才覆寫本機資料。
5. 遷移失敗：保留原始資料並提示匯出，不可直接清除。

## 10. 隱私與安全

- 所有資料預設只存在目前瀏覽器。
- 不把個案資料放入 URL、分析事件、錯誤紀錄或 console。
- GitHub 只能保存程式與測試用假資料，不得保存真實個案。
- 匯出的 JSON 為明文，畫面必須清楚警示。
- P5 才評估加密與跨裝置同步。

## 11. 介面規劃

新增「個案管理（僅此裝置）」卡片：

- 搜尋框
- 個案列表
- 儲存為新個案
- 覆寫目前個案
- 載入
- 刪除
- 匯出全部
- 匯入備份
- 本機筆數與最後備份時間

手機版按鈕使用單欄或雙欄排列，不可造成水平捲動。

## 12. 自動測試

至少包含：

1. 空資料庫初始化。
2. 新增一筆。
3. 同名新增兩筆，不互相覆寫。
4. 覆寫保留 `createdAt`。
5. 刪除指定 ID。
6. 搜尋姓名。
7. 搜尋生日。
8. 修改時間排序。
9. 匯出再匯入完整一致。
10. 無效 JSON 被拒絕。
11. 不支援的 schema 被拒絕。
12. 匯入失敗不改變原資料。
13. 100 筆資料保存、讀取與搜尋正常。
14. 讀取舊 schema 可遷移。

## 13. 人工驗收

- Safari 一般分頁保存後重開仍存在。
- 加入主畫面的 PWA 保存後重開仍存在。
- 清除網站資料後資料會消失，畫面事前有備份提醒。
- 私密瀏覽模式不可保證長期保存，畫面應提示。
- 匯出檔可在 iPhone「檔案」App 找到。
- 匯入後可重新計算且結果與目前引擎一致。

## 14. 開發順序

1. 建立獨立 `case-store.js`，不混入公式核心。
2. 完成純資料層自動測試。
3. 加入個案管理 UI。
4. 串接表單載入與重新計算。
5. 完成 JSON 匯出與匯入。
6. 完成 100 筆壓力測試與 iPhone 人工驗收。
7. P1 穩定後才開始 P2 報告草稿保存。
