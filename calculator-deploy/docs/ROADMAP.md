# 靈魂萬花筒計算器｜產品與技術路線圖

## 原則

- 先鎖定公式，再做資料保存。
- 先做本機資料，再做雲端資料。
- 報告、解讀與靈魂萬花圖必須共用同一核心計算結果。
- 每一階段都要有版本、驗收條件與回復方式。
- 正式來源檔必須可直接閱讀，不允許再依賴建置腳本臨時改寫 HTML、JS、CSS 或快取版號。

## P0｜核心計算穩定化

目標版本：2.2.x  
狀態：主要工程完成，等待人工驗收與試算表同步

已完成：

- 國曆／農曆換算與跨年生日判斷。
- 凌晨 12 點使用 24。

- 27 項核心回歸測試。

- 規則鎖定表與技術稽核文件。
- 首頁、輸入卡片與結果卡片共用同一外層寬度。
- 國曆與農曆生日狀態合併為同一格顯示。
- 正式來源檔直接維護版號、版面、DOM 與快取；建置不再改寫 HTML、JS、CSS、SW。
- 建置新增靜態來源檢查、JavaScript 語法檢查、版本一致性檢查與公式回歸門檻。

待完成：

- 將整合試算表的兩項舊口徑同步修正。

完成門檻：

- Cloudflare 組建紀錄顯示靜態來源檢查與 27 項公式測試全部通過。

- 整合試算表與 `RULES_LOCK.md` 不再存在已知規則衝突。

## P1｜個案本機儲存、搜尋與備份

目標版本：2.3.x  
狀態：技術規格已建立，尚未進入正式功能開發

功能：

- 儲存目前個案到瀏覽器本機。
- 個案列表、姓名搜尋與最近使用排序。
- 載入、覆寫、另存、刪除。
- JSON 匯出備份與 JSON 匯入還原。
- 匯出單一個案文字或長圖。
- 每筆紀錄保存資料格式版本、計算器版本與建立／修改時間。
- 匯入時先預覽筆數、重複資料與格式錯誤，再決定合併或取代。

資料原則：

- 預設不送到伺服器。
- 不在網址參數保存姓名、生日或出生時間。
- 不在 GitHub、Cloudflare 日誌或錯誤訊息輸出完整個案資料。
- 儲存「輸入資料」為主，載入後由目前核心引擎重新計算，避免永久保存過期結果。
- 清除瀏覽器資料前提醒先匯出備份。

完成門檻：

- 可儲存至少 100 筆個案。
- 匯出後清空，再匯入可完整還原。
- 舊版資料結構可遷移，不因改版遺失。
- 重複姓名不會互相覆蓋，每筆個案使用獨立 ID。
- 匯入錯誤檔案不會破壞既有資料。

詳細規格：`calculator-deploy/docs/P1_STORAGE_SPEC.md`

## P2｜個案報告產生器與 PDF

目標版本：3.0.x

功能：

- 基本資料與國／農曆計算摘要。
- 靈魂主數、靈魂等級、貴人、木馬、流年／流月／流日。
- 報告段落可選擇開啟或關閉。
- 可編輯草稿、預覽、輸出 PDF。
- 報告頁尾記錄引擎版本、資料庫版本與產生日期。

完成門檻：

- 手機與桌面預覽一致。
- PDF 不截字、不溢出、不遺漏頁面。
- 同一個案重算後可產生差異摘要。

## P3｜0～9 解讀、大師數與特殊數字資料庫

目標版本：3.1.x

資料結構：

- 數字主題。
- 成熟展現。
- 失衡提醒。
- 關係表現。
- 工作／財富／生活應用。
- 行動建議。
- 客戶報告文字。
- 來源與版本。

範圍：

- 0～9 基礎數字。
- 10、11、12 等中間鏈條。
- 大師數與特殊組合。
- 靈魂等級語句。
- 流年、流月、流日與位格解讀。

完成門檻：

- 每條文字均有來源、版本與適用欄位。
- 避免斷言、標籤與醫療化措辭。
- 程式只引用已核准內容。

## P4｜靈魂萬花圖核對與提示詞生成

目標版本：3.2.x

功能：

- 自動產生固定核對表。
- 判斷生日前版與生日後版。
- 套用固定九宮格、同心圓與幾何原型。
- 輸出少字精準版生圖提示詞。
- 保留主命數、日月綻放、內頻、木馬、貴人、流年與今年位格。
- 今年位格圖面只顯示最後主數。

完成門檻：

- 出圖前必須通過位置與數字核對。
- 外圈可隨年度變化，核心主頻不得改動。
- 圖面配色只使用數字對應色，不使用彩油瓶顏色。

## P5｜帳號、雲端同步、隱私與權限

目標版本：4.0.x

前置條件：P1、P2 穩定後才開始。

功能：

- 登入與跨裝置同步。
- 個案資料加密。
- 備份、還原與刪除機制。
- 權限分級：管理者、療癒師／顧問、只讀。
- 操作紀錄與資料版本。

完成門檻：

- 有隱私政策、資料刪除流程與備份策略。
- 個案敏感資料不得寫入公開 GitHub、網址或日誌。
- 權限測試與資料外洩風險檢查通過。

## P6｜正式 iOS／Android App

啟動條件：

- PWA 已有穩定使用者與明確離線、通知或裝置功能需求。
- App Store 成本與維護責任已確認。

建議路徑：

1. 先維持 PWA。
2. 有需求時使用 Capacitor 封裝。
3. 最後才評估原生 Swift／Kotlin。

不建議現在直接重寫原生 App，會分散公式校準與內容資料庫的資源。

## 建議執行順序

1. 完成 P0 人工驗收與試算表同步。
2. 開發 P1 本機個案庫與備份。
3. 建立 P3 解讀資料庫骨架。
4. 完成 P2 報告輸出。
5. 整合 P4 靈魂萬花圖。
6. 最後處理 P5 雲端同步與 P6 商店上架。
