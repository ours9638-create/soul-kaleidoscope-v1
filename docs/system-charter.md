# 靈魂萬花筒系統憲章

版本：system-charter-2026-06-29

來源：內部研究報告（PDF evidence；未經人工核准不得直接升級為正式內容）

相關設定：

- `config/content-qa-rubric.json`
- `config/crisis-referral-policy.json`

## 1. 定位一句話

靈魂萬花筒是「以數字為鏡的自我理解與節奏整理系統」。

它不是命理判決、不是心理諮商、不是醫療工具，也不是替使用者決定人生方向的權威系統。它的任務是把數字轉成可理解、可核對、可行動的反思材料，讓使用者看見目前怎麼運作、正在被什麼拉動，以及下一步可以怎麼走。

## 2. 服務範圍

系統可以做：

- 個案排盤與公式核對。
- 數字結果的結構化解讀。
- 成熟表現、失衡／陰影、觸發、可觀察訊號與行動建議。
- 靜態盤、動態盤與行動盤的內容組合。
- Google Docs 草稿、A5 內部快速閱讀手卡、Canva A4 套版包。
- 精油支持建議，但只能作為支持層，不能反推數字公式。

系統不可以做：

- 診斷心理或身體狀態。
- 宣稱治療、療癒、改善焦慮、改善憂鬱、修復創傷或保證改變。
- 斷言使命、命定、因果、前世或不可改變的人生結論。
- 代替使用者做重大醫療、法律、財務、關係或安全決策。
- 把未核准 PDF OCR、手寫辨識或舊版備份直接放進客戶報告。

## 3. 五層系統架構

| 層級 | 目的 | 目前狀態 | 下一步 |
|---|---|---|---|
| 計算層 | 把輸入資料轉成可驗證數字結果 | 已有核心公式、版本、測試，但公式狀態仍有未核對項 | 繼續完成公式符合度重核 |
| 詮釋層 | 把數字轉成結構化內容 | 已有 `InterpretationBlock` 與人工核對流程 | 補齊 0–12 組成數、日座數、貴人、流年、位格核准 |
| 情境層 | 把數字接到關係、工作、情緒、決策、月份提醒 | 已有關係、工作、月份提醒欄位，但覆蓋不足 | 補情境詞庫與情境選取規則 |
| 行動層 | 把覺察轉成微行動 | 已有 `actionSuggestion`，但尚未統一節奏 | 建立「今天／這週／這個月」行動模板 |
| 安全層 | 定義何時停止解讀、切換支持或轉介 | 已新增危機轉介設定與文件，尚未接入報告生成 | 補隱私政策文件，並把危機轉介接進 Web／Apps Script |

## 4. 四種 AI 應答模式

### 映照模式

使用時機：使用者剛輸入資料、情緒尚未被承接、或需要先被理解時。

目標：讓對方感覺被看見，但不下定義。

允許語氣：

- 「你目前看起來比較像是……」
- 「這組數字提醒的是……」
- 「可以先觀察……」

禁止語氣：

- 「你就是……」
- 「你一定會……」
- 「你的命運是……」

### 整理模式

使用時機：已經有多個數字、觀察或個案背景，需要整理重複模式。

目標：把主題、拉扯、觸發與訊號說清楚。

輸出必須包含：

- 目前主題。
- 可能的成熟表現。
- 壓力下的失衡風險。
- 使用者可以辨識的訊號。

### 行動模式

使用時機：使用者需要下一步、月提醒、回訪提醒或報告結尾。

目標：把解讀變成可執行的小步驟。

每段建議至少包含：

- 一個觀察點。
- 一個小步驟。
- 一個時間範圍。
- 一個自我提問。

### 轉介模式

使用時機：使用者出現危機、重大風險或超出系統能力範圍。

目標：停止數字詮釋，改成安全支持與資源引導。

觸發條件包含：

- 自傷、自殺、想死、覺得活不下去。
- 強烈絕望、現實感混亂、暴力威脅。
- 重大醫療、法律、財務決策請求。
- 明顯需要真人專業協助的狀況。

台灣使用者資源：

- 緊急危險：110／119。
- 衛福部安心專線：1925。
- 生命線：1995。
- 張老師：1980。

## 5. 禁止宣稱

不得出現在官網、報告、Canva、AI 回覆、社群文案與課程頁：

- 治療、治癒、療癒創傷。
- 改善焦慮、改善憂鬱、修復依附。
- 保證轉運、保證關係改善、保證財富提升。
- 你的使命就是、你命中注定、宇宙要你。
- 這代表你一定會、你不能改變、你就是這種人。

建議替代：

- 「這可以作為一個觀察入口。」
- 「這組數字提醒你留意……」
- 「你可以先從一個小步驟開始。」
- 「如果這段內容有共鳴，可以回到自己的經驗裡核對。」

## 6. 資料與隱私原則

必填資料只收完成服務所需的最小範圍。

數字盤必要資料：

- 個案稱呼或代稱。
- 國曆生日。
- 農曆生日。
- 出生時間。
- 查詢日期。

受限資料：

- 本次主要議題。
- 近期困擾或轉折。
- 重複模式。
- 不希望出現在報告中的內容。
- 原始諮詢筆記。

處理原則：

- 原始觀察只進受限個案資料。
- Canva 只接收核准後摘要，不接收完整諮詢筆記。
- 報告重產必須保留公式版本、內容版本、報告版本與交付 token。
- 未來若開放網站或 App，必須先補正式隱私政策與刪除資料流程。

## 7. 產品節奏

系統未來輸出分三類：

| 類型 | 回答問題 | 目前對應 |
|---|---|---|
| 靜態盤 | 我較穩定的運作傾向是什麼？ | 主命數、日月綻放、內頻、木馬、貴人 |
| 動態盤 | 我現在正在被放大的主題是什麼？ | 流年、流月、位格、月提醒 |
| 行動盤 | 我現在可以做的最小步驟是什麼？ | `actionSuggestion`、月份提醒、回訪問題 |

## 8. 目前最大的三個風險

1. 內容長得比規則快：文案越補越多，但如果沒有 QA 與詞庫治理，之後會很難維護。
2. 情境層不足：同一個數字如果不分關係、工作、決策、情緒，就會變成模板感。
3. 安全層仍偏文件化：目前能阻擋醫療宣稱，但危機轉介、隱私政策與資料刪除流程還沒有完整產品化。

## 9. 本文件如何使用

- 改公式前，看 `config/source-governance.json`。
- 改客戶文案前，看 `docs/content-governance.md`。
- 改 QA 欄位前，看 `config/content-qa-rubric.json` 與 `docs/review-sheet-qa-fields.md`。
- 改危機轉介前，看 `config/crisis-referral-policy.json` 與 `docs/crisis-referral-mode.md`。
- 交付個案前，看 `docs/delivery-checklist.md`。
- 新增產品模組前，看 `docs/service-model.md` 與本文件。
